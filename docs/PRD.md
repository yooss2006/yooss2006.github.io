# PRD: 안개비 (`rainy-notes`)

## 제품 요약

- 프로젝트명: `rainy-notes`
- 블로그명: **안개비**
- 배포 전제: GitHub Pages 기반 정적 사이트
- 핵심 목적: 개발 글과 일상 글을 함께 오래 남기고픈 **개인 기록장**
- 핵심 경험: **“비는 배경이고, 글이 중심이다.”**
- 시각 톤: 미니멀 텍스트 우선
- 분위기 요소: 배경에 비가 은은하게 내리고, 사용자가 원할 때 빗소리를 들을 수 있음

## 제품 원칙

### 글 우선

모든 시각 효과와 소리 효과는 글 읽기를 보조해야 한다. 비 애니메이션과 빗소리가 블로그의 정체성을 만들 수는 있지만, 본문을 방해해서는 안 된다.

### 정적 사이트 우선

GitHub Pages에 자연스럽게 배포할 수 있는 정적 사이트 구조를 우선한다. 서버, 데이터베이스, 로그인, 관리자 페이지는 첫 버전에서 제외한다.

### 가벼운 기록 흐름

블로그의 핵심은 꾸준히 쓰는 것이다. 글 하나를 작성하고 게시하는 흐름이 무거워지면 제품 목적과 충돌한다.

### 접근성 기본값

모션과 소리는 사용자가 통제할 수 있어야 한다. 특히 비 애니메이션은 `prefers-reduced-motion` 설정을 존중해야 하며, 빗소리는 기본적으로 자동 재생하지 않는 방향을 권장한다.

### 방향과 구현 분리

이 문서는 제품 방향과 첫 구현 기준을 함께 정리하는 단일 PRD다. 첫 구현 스택은 Astro 기반 정적 블로그로 확정한다.

## 범위

### 포함 범위

첫 제품 방향에는 다음을 포함한다.

- GitHub Pages에 배포 가능한 정적 블로그 방향
- 개발 글과 일상 글을 함께 담는 개인 기록장 경험
- 홈, 글 목록, 글 상세, 소개 페이지 중심의 기본 정보 구조
- 배경 비 애니메이션 방향
- 빗소리 사용 경험 방향
- 페이지 이동 중에도 끊기지 않는 빗소리 지속 경험
- 접근성 및 가독성 원칙
- 미니멀 텍스트 우선 시각 톤
- 후속 구현 단계에서 사용할 수 있는 검증 기준

### 제외 범위

첫 버전에서는 다음을 목표로 하지 않는다.

- 채용/브랜딩/프로젝트 쇼케이스 중심 포트폴리오 최적화
- 광고, 후원, 유료 구독, 뉴스레터 판매 등 수익화
- 댓글, 로그인, 좋아요, 팔로우 등 커뮤니티 기능
- 서버형 CMS 또는 관리자 페이지
- Obsidian급 지식 그래프, 백링크, 완전한 지식관리 시스템
- GitHub 원격 설정, 실제 배포, 커스텀 도메인 설정
- 서버 런타임이 필요한 실제 운영 기능

## 사용자

### 주요 사용자

- **누구**: 개발 글과 일상 글을 오래 남기고 싶은 글쓴이 본인
- **상황**: 새 글을 쓰고, 기존 글을 다시 읽고, 공개 방문자에게 기록장을 보여주는 상황
- **목표**: 포트폴리오보다 개인 기록의 흐름을 가볍게 유지하는 것

### 방문자

- **누구**: 글쓴이의 개발 기록이나 일상 기록을 읽는 공개 방문자
- **상황**: 홈에서 최근 글을 보거나, 글 목록에서 관심 있는 글을 찾거나, 공유 링크로 글 상세에 진입하는 상황
- **목표**: 장식에 방해받지 않고 글을 편하게 읽는 것

## 정보 구조

비 애니메이션과 소리 토글은 컨텐츠 영역을 침범하지 않는다.
첫 버전의 기본 구조는 다음을 권장한다.

### 홈

- 짧은 소개를 보여준다.
- 최근 글을 최대 5개 보여준다.

### 글 목록

- 전체 글을 날짜순으로 보여준다.
- 제목, 날짜, 썸네일, 짧은 요약을 표시한다.

### 글 상세

- 본문 가독성을 최우선으로 한다.
- 읽는 동안 시각적 변화가 과하지 않아야 한다.

### 소개

- 블로그의 목적과 글쓴이의 짧은 소개를 보여준다.
- 채용용 포트폴리오 페이지처럼 과하게 정제하지 않는다.

## 사용자 경험 방향

### 시각 경험

- 미니멀 텍스트 우선
- 넓은 여백
- 낮은 시각적 밀도의 비 효과
- 낮은 대비의 배경 장식
- 본문과 배경 효과의 명확한 분리
- 모바일에서도 본문 폭, 행간, 여백 우선

### 비 애니메이션

비는 블로그의 정체성을 만드는 요소지만, 본문보다 앞서면 안 된다.

권장 방향:

- 낮은 투명도
- 느린 움직임
- 본문 영역과 겹쳐도 가독성을 해치지 않는 대비
- 모션 줄이기 설정이 켜진 경우 정지 또는 최소화

### 빗소리

빗소리는 분위기를 만드는 요소지만, 자동으로 사용자를 방해해서는 안 된다.

권장 방향:

- 기본값은 꺼짐
- 사용자가 직접 켤 수 있음
- 쉽게 끌 수 있는 토글 제공
- 가능하면 사용자의 선택 상태를 유지
- 내부 페이지 이동 중 재생이 끊기지 않음
- 갑작스럽거나 반복적인 소리 변화 금지

구현 방향:

- Astro 기본 페이지 이동은 일반 브라우저 전체 페이지 이동이므로 오디오가 끊길 수 있다.
- Phase 1에서는 공통 레이아웃에 Astro `ClientRouter`를 적용하고, 오디오 플레이어를 `transition:persist` 대상 컴포넌트로 구현한다.
- 빗소리 지속은 사용자가 명시적으로 켠 뒤에만 적용한다.
- 클라이언트 라우팅을 적용하므로 페이지 이동 후 스크립트 재초기화, 포커스, 뒤로 가기, 접근성 안내를 함께 검증한다.

## 접근성 원칙

- `prefers-reduced-motion` 설정을 존중한다.
- 소리는 자동 재생하지 않는다.
- 소리 토글은 키보드로 접근 가능해야 한다.
- 본문 색 대비를 우선한다.
- 비 효과가 글자 위에 올라와도 가독성을 해치지 않아야 한다.
- 장식 요소 때문에 스크롤, 클릭, 읽기 흐름이 방해되면 안 된다.
- 클라이언트 라우팅을 사용하더라도 페이지 제목, 포커스 이동, 키보드 탐색이 깨지면 안 된다.

## 사용자 흐름

### 핵심 흐름

```text
홈 -> 최근 글 선택 -> 글 상세 읽기
홈 -> 글 목록 -> 관심 글 선택 -> 글 상세 읽기
어느 페이지든 -> 빗소리 켜기 -> 내부 페이지 이동 -> 빗소리 유지
```

### 상세 흐름

1. **홈 진입**: 방문자는 블로그명, 짧은 소개, 최근 글을 본다.
2. **글 탐색**: 방문자는 최근 글 또는 글 목록에서 읽을 글을 고른다.
3. **글 읽기**: 글 상세에서는 본문 폭, 행간, 여백이 최우선이다.
4. **빗소리 사용**: 사용자가 직접 토글을 켜면 빗소리가 재생된다.
5. **페이지 이동**: 내부 링크로 이동해도 빗소리는 끊기지 않는다.

## AI 구현 규칙

### 절대 하지 마

- 서버, DB, 로그인, 관리자 페이지를 만들지 않는다.
- 빗소리를 자동 재생하지 않는다.

- 새 글 작성에 코드 수정을 요구하는 구조로 만들지 않는다.
- GitHub Pages `base` 경로 문제를 무시하지 않는다.
- 클라이언트 라우팅 적용 후 페이지 이동 접근성 검증을 생략하지 않는다.

- 빗소리 오디오 플레이어는 공통 레이아웃에서 지속 가능한 구조로 둔다.
